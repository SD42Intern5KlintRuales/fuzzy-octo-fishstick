Activity Title
Release Document Required Field Validation Tool

Objective
Create a web-based automation tool that validates uploaded Excel files and checks whether required fields contain inputs.

The tool should not validate every cell in the Excel file. It should only validate the required fields configured in the validation rules.

There are 2 Excel files that need to be supported:
1. Release Notes
2. Test Specification/Result

Current Sample File Used
File Name: Support SC077-51 Final Release Notes.xlsx
Detected Sheets:
- ReleaseNote
- Product_ReleaseCheckpoints
- ReleaseDetailsHistory

Main Validation Concept
The tool should check if required cells or required row fields have content.

Example:
If the required field is ReleaseNote!R3, the tool should check if R3 has a value.

If R3 has a value:
Result: PASS

If R3 is blank:
Result: FAIL
Message: Missing required input for Approved By

Important Rule
The validation should be configurable.
Avoid hardcoding all required cells directly in the source code.
Use a JSON configuration file so that required fields can be updated easily.

Recommended Validation Rules for Release Notes File

1. Sheet: ReleaseNote
Required Fields:
- Project Name
  - Cell: E2
  - Expected check: Must not be blank

- Release Purpose
  - Cell: E3
  - Expected check: Must not be blank

- Release Date
  - Cell: E4
  - Expected check: Must not be blank and should be a valid date

- Prepared By
  - Cell: L3
  - Expected check: Must not be blank

- Confirmed By
  - Cell: O3
  - Expected check: Must not be blank

- Approved By
  - Cell: R3
  - Expected check: Must not be blank

- Branch Name
  - Cell: B12
  - Expected check: Must not be blank

- Version
  - Cell: L12
  - Expected check: Must not be blank

- Changes
  - Cell: B14
  - Expected check: Must not be blank

- Supported Languages
  - Cell: B16
  - Expected check: Must not be blank

- Supported Machine(s)/Environment(s)
  - Cell: B18
  - Expected check: Must not be blank

- Limitations
  - Cell: B20
  - Expected check: Must not be blank

Notes:
- Values such as N/A or - may be accepted as valid input if the field is intentionally not applicable.
- The tool should only fail the field if the cell is empty, null, or contains only spaces.

2. Sheet: Product_ReleaseCheckpoints
Required Fields:
- Category
  - Column: B
  - Starting Row: 4
  - Expected check: Must not be blank for checkpoint rows

- Checkpoint
  - Column: C
  - Starting Row: 4
  - Expected check: Must not be blank

- Results
  - Column: D
  - Starting Row: 4
  - Expected check: Must not be blank

Recommended dynamic row validation:
- Start checking from row 4.
- Continue checking rows while Category, Checkpoint, or Result contains data.
- If a row has a Checkpoint but no Result, mark it as FAIL.
- If a row has a Result but no Checkpoint, mark it as FAIL.
- If Category is only written once for a group, the tool may allow blank Category cells in the next rows, as long as Checkpoint and Result are filled.

Example from sample file:
- B4: Codebase
- C4: ServiceEngine should build successfully.
- D4: ✓
- C5: JaCoCo coverage should be 60% up
- D5: ✓
- C6: All Unit Tests should pass
- D6: ✓

3. Sheet: ReleaseDetailsHistory
Required Fields:
- Release Date
  - Cell: C3
  - Expected check: Must not be blank and should be a valid date

- Version
  - Cell: C4
  - Expected check: Must not be blank

- Implemented function(s)/Fixed defect(s)
  - Cell: C5
  - Expected check: Must not be blank

- Known defect(s)/limitation(s)
  - Cell: C7
  - Expected check: Must not be blank

- Note(s)
  - Cell: C8
  - Expected check: Must not be blank

Notes:
- If implemented functions span multiple rows, the tool should check that at least the first configured cell has content.
- Optional enhancement: combine C5 to C6 or detect all rows between Implemented function(s)/Fixed defect(s) and Known defect(s)/limitation(s).

Validation Logic Summary
1. Upload the required Excel files.
2. Identify the document type.
3. Check if required sheets exist.
4. Load validation rules from the JSON configuration file.
5. For each configured cell rule:
   - Read the cell value.
   - Trim spaces.
   - If the value is empty, mark as FAIL.
   - If the value exists, mark as PASS.
6. For each configured table rule:
   - Start from the configured row.
   - Read each row until the table ends.
   - Validate required columns.
   - Add PASS or FAIL result per row field.
7. Return a structured validation summary to the frontend.
